Learn Python the hard way-Ex43 基本的面向对象的分析和设计

2019-07-01

自上而下的执行方式：

1. 先设计出框架：
画图表或流程图

2. 再提炼关键问题:
根据图表/流程图，列出所有的名词和动词，写出它们是如何相关的。

3. 然后根据问题创建有层次结构的类和对象映射:
创建简单的树来说明概念之间的关系（名词和动词），比如确定概念的子父类关系，直到得到一个有层次结构的类（树状图）。检查这个树状图的概念关系清晰。

4. 最后编写代码并测试:
写基本框架代码，代码中先只包括之前提到的各种类和每个类中包含的函数。然后用测试例子逐段测试代码。一定要逐段测试，不要一起写完再一起测试，那样不容易发现问题到底出在哪里。

5. 精简代码。

ex43.py会是一个叫做“Gothons from Planet Percal #25”的游戏，我们按照上述的步骤来构建游戏并写出对应代码。

创建有层次结构的类和对象映射的概念：
把名词列表转换成一个有层次结构的类树。
比如：

  • ”Room”和”Scene”对我来说要做的事情是一样的。在举例中作者选择了”Scene”，所有的特殊房间比如”Central Corridor”基本上也跟”Scene”一样。”Death”也是一个”Scene”，所以游戏中展现的是一个“死亡场景”而非“死亡房间”。
  • “Maze”和”Map”基本一样，作者选择用”Map”。
  • 不想做战斗系统，所以先忽略”Alien”和”Player”，但保存下来以后使用。
  • “Planet"也可能只是一个场景，不是什么具体事情。
    所以梳理下来：
  • Map
  • Engine
  • Scene
    • Death
    • Central Corridor
    • Laser Weapon Armory
    • The Bridge
    • Escape Pod

确认每个动词需要什么行动。
比如从描述中知道，需要一个方法来“run”这个引擎，通过地图获得下一个场景“get the next scene”，获得“opening scene”，或者“enter”一个场景。把这部分添加到上面的类树中：

  • Map - next_scene - opening_scene
  • Engine - play
  • Scene - enter * Death * Central Corridor * Laser Weapon Armory * The Bridge * Escape Pod
    ⚠️注意，这里把-enter放在Scene下面，因为知道所有的场景都会继承它并重写它。

🤔几个疑惑：
pass什么意思？
python中的pass是空语句，是为了保持程序结构的完整性。pass不做任何事情，一般用作占位语句。
举例：

#!/usr/bin/python
#-*- coding: UTF-8 -*-
#输出python的每个字母
for letter in ‘Python’:
    if letter == ‘h’:
        pass
        print(‘这是pass快’)
    print(‘当前字母：’, letter)
print(“Good bye!”)
结果：
当前字母：P
当前字母：y
当前字母：t
这是pass块
当前字母：h
当前字母：o
当前字母：n
Good bye!

除了自上而下的方法，还有自下而上的方法构建代码。自上而下的方法适合于已经知道整体构架，需要进行细化并执行的时候；自下而上则适用于暂时只看到局部需要解决的问题，还没有足够的信息把握整个概念。自下而上的方法如下：

  1. 取一小块问题，编写一些代码，让它们勉强运行；
  2. 完善代码，将其转换成一些更正式的包含类和自动化测试的代码；
  3. 提取其中的关键概念，并尝试找出研究它们；
  4. 写出到底发生了什么的描述；
  5. 继续完善代码，也可能是把它们扔掉，并重新开始；
  6. 移动到其他问题上，重复步骤。

class Engine(object):

    def __init__(self, scene_map):
        self.scene_map = scene_map

    def play(self):
        current_scene = self.scene_map.opening_scene()
        last_scene = self.scene_map.next_scene('finished')

        while current_scene != last_scene
            next_scene_name = current_scene.enter()
            current_scene = self.scene_map.next_scene(next_scene_name)

        #be sure to print out the last scene_map
        current_scene.enter()

from sys import exit  #导入exit模组
from random import randint  #导入randint模组
from textwrap import dedent  #导入dedent模组

class Scene(object):  #创建Scene场景类

    def enter(self):  #创建Scene下enter函数
        print("This scene is not yet configured.") #enter函数包括两个打印
        print("Subclass it and implement enter().")
        exit(1)  #exit(1是有错误退出/异常退出

class Engine(object): #创建Engine类

    def __init__(self, scene_map):  #创建Engine自己的函数scene_map
        self.scene_map = scene_map

    def play(self):  #创建play
        current_scene = self.scene_map.opening_scene()  #当前场景类的实例化， current_scene就是当前场景类名()
        last_scene = self.scene_map.next_scene('finished')  #创建一个胜利结束场景类对象，准备好下面进入关键的循环阶段

        while current_scene != last_scene:  #如果游戏没有胜利（当前场景非胜利场景），进入循环后先进行目前场景的游戏部分，.enter()运行，结束后返回的是下一个场景的key
            next_scene_name = current_scene.enter()
            current_scene = self.scene_map.next_scene(next_scene_name)
            #一个场景enter结束后要开始进入下一个场景，下一个场景类的实例化。如果.enter()进入Death场景，无返回值，游戏结束
        #be sure to print out the last scene_map
        current_scene.enter()  #current_scene == last_scene，所以是last_scene.enter()，结束循环，游戏胜利

class Engine(object):
    def __init__(self, scene_map):
        self.scene_map = scene_map

    def play(self):
        current_scene = self.scene_map.opening_scene()
        last_scene = self.scene_map.next_scene('finished')

        while current_scene != last_scene:
            next_scene_name = current_scene.enter()
            current_scene = self.scene_map.next_scene(next_scene_name)

        current_scene.enter()

  1. 这是一个小版本的 “有限状态机器finite state machine” 。找资料了解一下。
    Finite-state machine, FSM，又称有限状态自动机，简称状态机。表示有限个状态以及在这些状态之间的转移和动作等行为的数学模型。FSM是一种算法思想，由一组状态、一个初始状态、输入和根据输入及现有状态转换为下一个状态的转换函数组成。现实世界存在大量具有有限个状态的系统：钟表系统、电梯系统、交通信号灯系统等。
    ** FSM有多种实现方式：**
    ** switch-case或if-else**
    比如RPG游戏中的城门这个对象，具有opened，closed，locked，unlocked四种状态。当玩家到达一个locked的门时，如果玩家已有开门的钥匙，那么就可以顺利让门的状态变为unlocked，进一步可以使之变成opened。
switch (state)  {
  // 处理状态Opened的分支
  case (Opened): {
    // 执行动作Open
    open();
    // 检查是否有CloseDoor事件
    if (closeDoor()) { 
      // 当前状态转换为Closed
      changeState(Closed)
    }
    break;
  } 
  // 处理状态Closed的分支
  case (Closed): {
    // 执行动作Close
    close();
    // 检查是否有OpenDoor事件
    if (openDoor()) {
      // 当前状态转换为Opened
      changeState(Opened);
    }
    // 检查是否有LockDoor事件
    if (lockDoor()) {
      // 当前状态转换为Locked
      changeState(Locked);
    }
    break;
  }
 
  // 处理状态Locked的分支
  case (Locked): {
    // 执行动作Lock
    lock();
    // 检查是否有UnlockDoor事件
    if (unlockDoor()) {
      // 当前状态转换为Unlocked
      changeState(Unlocked);
    }
    break;
  }
 
  // 处理状态Unlocked的分支
  case (Unlocked): {
    // 执行动作Unlock
    unlock();
    // 检查是否有LockDoor事件
    if (lockDoor()) {
      // 当前状态转换为Locked    
      changeState(Locked)
    }
    // 检查是否有OpenDoor事件    
    if (openDoor()) {
      // 当前状态转换为Opened
      changeSate(Opened);
    }
    break;
  }
